Where to Begin After Login

When an admin or manager logs in, the portal redirects automatically to the Admin Landing Page (/dashboard/admin/landing). This page is the primary action queue and the correct starting point for every decision session.

2

Understanding the Data Before You Decide

Before approving or denying any request, a manager must read three pieces of information that the portal surfaces on every decision row.

2.1 Working Priority (WP)

Working Priority is a manually recalculated rank assigned by the pre-processing staff — not an automated system output. Every six months, the pre-processing team reviews all submitted requests, applies the five priority rules, and imports the resulting WP values into the portal. Between recalculation cycles, the portal falls back to the employee's self-reported Priority (P) value. WP is therefore a human-reviewed, deliberate override of the raw submission data, and managers should treat it as the authoritative ranking signal when it is present.

When no WP has been assigned (i.e., the request was submitted after the most recent pre-season computation), the portal displays the employee's self-reported P value instead. In that case, apply the standard three-rule framework using P as the ranking input.

BadgeMeaning
WP1Employee's top-ranked request; highest claim on the date
WP2Second-ranked request
WP3–4Mid-tier requests
WP5+Lower-ranked; employee has higher-priority requests already in the system
ℹ️ P5 is the form default. Priority 5 is pre-filled on the New Request submission form. A P5 submission may mean the employee never set a preference — not that they ranked the request fifth. Pre-processing staff treat a lone P5 as WP1. Do not penalize a P5 request solely because of the number.

2.2 Seniority Rank (SR)

Seniority Rank is the unit-wide rank of the employee by their official seniority date (earliest date = most senior = rank 1). When two requests share the same Working Priority on the same date, the more senior employee (lower SR number) takes precedence.

2.3 Slot Usage and the 8-Person Cap

Each shift (AM, PM, NOC) has a hard ceiling of 8 approved vacation slots per calendar day. The portal displays slot usage as a fraction (e.g., 5 / 8) on every date row. Approving a ninth request requires a deliberate override and must be documented with a decision note.

2.4 Summer 14-Day Consecutive Cap

For July and August, dates beyond day 14 of a consecutive vacation run are flagged amber. These dates are not automatically denied — the admin retains full Approve / Deny / Clear authority — but the flag requires explicit manager judgment.

3

The Three-Rule Decision Framework

Every approval decision follows a strict three-rule hierarchy. Apply the rules in order; only move to the next rule if the current one does not produce a clear winner. This rubric makes more than 90% of first-choice approvals possible.

Rule 1 — Applied First
Working Priority

The request with the lowest WP number (WP1 beats WP2, WP2 beats WP3, etc.) is approved first. This rule resolves the majority of decisions without ambiguity.

Rule 2 — Tiebreaker
Seniority Date

When two requests share the same Working Priority on the same date, the employee with the earlier seniority date is approved. The portal displays the SR rank next to each name for immediate comparison.

Rule 3 — Soft Yield
21-Day Ceiling Yield

When an employee has already accumulated 21 or more approved vacation days in a period (Period A = Jan–Jun, Period B = Jul–Dec), their Priority 2 and above requests should yield to employees who have not yet reached 21 days, even if seniority would otherwise favor them. This is a soft yield — the manager makes the final call.

Ties & Gray Areas
Manual Admin Resolution

The portal does not auto-approve. Every decision is a human action with a full audit trail. Ties and gray areas are resolved manually by the admin.

Step-by-Step Approve / Deny Navigation

Path 1

Admin Landing Page — Daily Triage

  1. Log in. The portal lands on the Admin Landing Page automatically.
  2. In Section B (Pending Decision Dates), scan for dates with the highest pending counts or over-cap flags.
  3. Click any date row to open the Date/Shift Detail modal.
  4. Select the shift tab (AM, PM, or NOC) to view that shift's ranked request list.
  5. Read the WP badge, SR number, and slot usage for each row.
  6. Click Approve or Deny on each row. For denials, enter a denial note.
  7. The slot usage counter updates in real time. Repeat for all pending dates.
Path 2

Decision Board — Monthly Processing Runs

  1. Navigate to Administration → Decision Board in the sidebar.
  2. Select the target month (July 2026 – January 2027).
  3. Select the shift tab (AM, PM, or NOC).
  4. Left panel: all dates with per-date pending / approved / denied counts.
  5. Click any date to load ranked request rows in the right panel.
  6. Work top-to-bottom: Approve / Deny / Clear. The cap divider line shows the 8-person ceiling.
  7. Summer Cap rows (amber) require an explicit decision — they will not auto-resolve.
  8. Green checkmarks appear when all rows on a date are decided.
⏱ Target pace: One month per working day. The full July–December cycle should be completed within one week of the submission window closing.
Path 3

Date/Shift Detail Modal — Spot Decisions from Section A

  1. In Section A (Recent Requests), click the date link on any row.
  2. The Date/Shift Detail modal opens pre-filtered to that date and shift.
  3. Apply Approve / Deny / Clear as needed.

Clearing a Decision

Every approved or denied row has a Clear (undo) button — a circular arrow icon. Use Clear when a decision was made in error or when circumstances change (e.g., a competing request is withdrawn, freeing a slot). Clearing returns the row to pending and recalculates the slot usage counter.

Certain dates may be designated as blackout dates — days on which no vacation requests are accepted regardless of seniority or priority. Configured in Policy Settings → Blackout Dates. Common triggers: mandatory unit-wide training, Joint Commission survey windows, critically short staffing periods.

6

Announcing the Close of Submission Windows

Closing a submission window is a governance action, not just a system setting. It requires both a portal configuration change and a staff communication.

  • Set the Deadline in the Portal — 14 Days Before Close

    Navigate to Administration → Policy Settings → Submission Deadlines. Enter the deadline date, coverage start, and coverage end. Save. The portal immediately displays the deadline to employees on the New Request form and My Portal page.

  • Post an Announcement — 7 Days Before Close

    Navigate to Administration → Announcements. Create an announcement stating the exact deadline date and time (11:59 PM Pacific), the coverage period, and a reminder that no requests will be accepted after the deadline regardless of seniority.

  • Send a Direct Email — 3 Days Before Close

    Use the Message Super Admin feature or your unit email distribution list to send a final reminder directly to all staff. This step is a manual manager action.

  • Lock the Window — On the Deadline Date

    Verify in Policy Settings that the deadline is correctly recorded. The portal enforces the deadline automatically at midnight — no further action is required.

  • Begin the Approval Run — Within 5 Business Days of Close

    Start the formal approval run using the Decision Board. The standard commitment is decisions within 10 business days of the submission deadline.

Sample Announcement: "The submission window for the Winter Holiday Block (December 20 – January 5) closes on October 21, 2026 at 11:59 PM Pacific. All requests for dates in this period must be submitted before this deadline. Decisions will be communicated by November 15, 2026."

Governance Process

7.1 Who Decides

RoleDecision Authority
Manager / Charge NurseApproves or denies individual date-level requests using Approve / Deny / Clear controls
Admin (Anne Kiboi, Carol Robinson)Full access to all decision tools; can override manager decisions; manages deadlines, blackout dates, and announcements
Super Admin (Robert Domondon)Can add dates on behalf of any employee; receives escalated messages; final authority on all disputes

7.2 Decision Audit Trail

Every approve, deny, and clear action is recorded in the Audit Log with the actor's name, request ID, date, decision, and timestamp. Accessible at Administration → Audit Log. No decision can be made without leaving an audit entry — enforced at the database level.

7.3 Decision Note Policy

A decision note is required for any denial where the slot was available (within the 8-person cap). A decision note is strongly recommended for any approval that overrides the normal priority order. Notes are visible to the employee in their My Portal request history.

7.4 The PM Shift as a Staffing Equalizer

The PM shift acts as a staffing equalizer and smoother for both AM and NOC shifts. Most clinical procedures occur during the AM-PM overlap window. When reviewing PM shift requests, consider the downstream effect on AM and NOC coverage — a heavily approved PM date may require a compensating denial on an adjacent shift date.

7.5 Ancillary & Inactive Employees

Ancillary employees are excluded from all slot counts, calendar tallies, and approval run data. Inactive employees' requests are archived and excluded from all active counts. Both exclusions are enforced automatically across all portal views.

8

Escalation to Super Admin

When a manager or admin encounters a situation that exceeds their decision authority — a dispute between two employees of equal seniority and priority, a request to override a blackout date, or a staffing emergency requiring an exception — the correct escalation path is the Message Super Admin feature.

Navigate to Administration → Message Super Admin in the sidebar. The message is delivered to the Super Admin inbox and simultaneously sent by email to great.ai.nurses@gmail.com. The Super Admin can reply directly from the inbox.

🚫 Do not use personal email or text for escalations involving vacation decisions. The portal's messaging system creates a documented record of the escalation and the response, which is important for union compliance and grievance defense.

Related Topics

9.1 Period Day Count Tracking

Vacation usage is tracked in two periods: Period A (January–June) and Period B (July–December). The soft ceiling is 21 days per period. Employees approaching or exceeding 21 days are flagged in the 21-Day Ceiling Tracker (Administration → 21-Day Ceiling Tracker). Review this before beginning any approval run.

9.2 Education Requests

Education requests do not count against the 8-person vacation cap and do not appear in the vacation seniority-ranked lists. They have their own export report under Administration → Export Data → Education Report. The approval standard is generally more permissive than vacation, subject to shift coverage requirements.

9.3 Working Priority CSV and Pre-Season Computation

Before the annual vacation cycle opens, the Super Admin runs a working priority computation that assigns WP values to all submitted requests. Managers do not run this computation — it is a pre-season administrative step. WP values are fixed at import time and do not change unless the Super Admin manually updates them. For mid-cycle submissions, the portal automatically assigns working_priority = priority at submission time.

9.4 Weekend-3 Rotation Policy (Upcoming)

A revised weekend rotation policy is planned: nurses will work every other two weekends (Weekend-3 pattern). When this policy goes live, the portal will add a Rotation Group field (Group A, B, or C) to each employee record. Until the policy is formally activated, weekend requests are evaluated under the standard three-rule framework.

9.5 Daily Automated Backup

All portal data is automatically backed up at 2:00 AM Pacific every day to the private GitHub repository at AI-Nurse-Solutions/vnc-icu-portal-backup. This includes the full database dump, the last 24 hours of audit log entries, and a git diff summary of any code changes.

9.6 Browser Compatibility

Safari Private Browsing, Firefox with Strict ETP, and Brave with Aggressive Shields will block the session cookie and prevent login. Staff should use Chrome or Edge in standard (non-private) browsing mode.

9.7 Password and Account Management

Employees can use Forgot Password on the login page to receive a reset link (expires in 60 minutes). Admins can reset any employee's password from Administration → Employees. New self-registered employees are marked Unverified until an admin sets their official employee number and seniority date.
